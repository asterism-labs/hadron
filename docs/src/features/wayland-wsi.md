# Wayland Minimal Subset

## Goal

Implement the minimum Wayland protocol subset that Mesa's Vulkan WSI layer
requires, plus basic window management. This replaces the existing Lepton display
protocol with a Wayland-compatible compositor, enabling unmodified Mesa WSI code
to create surfaces and present frames.

## Background

Mesa's Vulkan WSI (Window System Integration) creates presentable surfaces via
the Wayland protocol. It links against `libwayland-client` and speaks the Wayland
wire protocol over Unix domain sockets with fd-passing. The compositor implements
the server side.

The existing Lepton compositor uses a custom 64-byte fixed-message protocol over
Hadron's channel IPC. Migrating to Wayland replaces this with a standard protocol
that Mesa and future applications can use directly.

## Key Design

### Protocol Objects

Mesa's `wsi_common_wayland.c` requires these protocol objects. Each is defined
by a Wayland XML protocol file and code-generated by `wayland-scanner`.

#### Core Protocol (`wayland.xml`)

| Object | Purpose | Required interfaces |
|--------|---------|---------------------|
| `wl_display` | Connection lifecycle | `get_registry`, `sync` |
| `wl_registry` | Global object discovery | `bind` |
| `wl_compositor` | Surface factory | `create_surface` |
| `wl_surface` | Drawable surface | `attach`, `damage`, `commit`, `frame` |
| `wl_shm` | Shared memory pool factory | `create_pool` |
| `wl_shm_pool` | Pixel buffer pool | `create_buffer`, `resize` |
| `wl_buffer` | Pixel buffer handle | `destroy` |
| `wl_callback` | One-shot event (frame sync) | `done` |

#### XDG Shell (`xdg-shell.xml`)

| Object | Purpose | Required interfaces |
|--------|---------|---------------------|
| `xdg_wm_base` | Window management base | `get_xdg_surface`, `pong` |
| `xdg_surface` | Window surface role | `get_toplevel`, `ack_configure` |
| `xdg_toplevel` | Top-level window | `set_title`, `set_app_id`, `close` |

#### DMA-buf (Phase 2, `linux-dmabuf-v1.xml`)

| Object | Purpose | Required interfaces |
|--------|---------|---------------------|
| `zwp_linux_dmabuf_v1` | DMA-buf buffer import | `create_params` |
| `zwp_linux_buffer_params_v1` | Buffer parameters | `add`, `create` |

### Transport

Wayland uses Unix domain sockets (`AF_UNIX`, `SOCK_STREAM`) with fd-passing via
`sendmsg`/`recvmsg` + `SCM_RIGHTS`. The compositor listens at
`$XDG_RUNTIME_DIR/wayland-0`. See [Unix Domain Sockets](unix-domain-sockets.md).

### Wire Protocol

The Wayland wire protocol uses a fixed header format:

```
┌──────────────┬──────────────┐
│  object_id   │ size | opcode│
│   (32-bit)   │ (16b)  (16b) │
├──────────────┴──────────────┤
│        arguments...         │
└─────────────────────────────┘
```

Arguments are 32-bit aligned: `int`, `uint`, `fixed` (24.8), `string`
(length-prefixed), `object` (ID), `new_id` (ID), `array` (length-prefixed),
`fd` (passed via SCM_RIGHTS ancillary data).

**libwayland-client** handles the wire protocol, object lifecycle, and event
dispatching. The port involves compiling it against hadron-libc with Unix domain
socket support — minimal patching expected if AF_UNIX is standards-compliant.

### Compositor Architecture

The compositor evolves from Lepton's design but speaks Wayland protocol:

```
┌──────────────────────────────────────────────────────┐
│                    Compositor                         │
│                                                      │
│  ┌──────────┐  ┌──────────┐  ┌────────────────────┐ │
│  │ Wayland  │  │ Window   │  │ Display Backend     │ │
│  │ Protocol │  │ Manager  │  │ (/dev/fb0 or        │ │
│  │ Server   │  │          │  │  VirtIO GPU scanout)│ │
│  └────┬─────┘  └────┬─────┘  └────────┬───────────┘ │
│       │              │                 │             │
│       └──────────────┴─────────────────┘             │
│                                                      │
│  UDS listener at /run/wayland-0                      │
│  Accept clients, dispatch protocol events            │
│  Composite surfaces, output to display               │
└──────────────────────────────────────────────────────┘
```

#### Rendering Paths

**Phase 1 (shm)**: Client renders to a shared memory buffer (`wl_shm_pool`),
compositor reads pixels via mmap, composites to the output framebuffer. This is
the lavapipe path — CPU rendering on both sides.

**Phase 2 (DMA-buf)**: Client renders to a GPU buffer, passes the DMA-buf fd to
the compositor via `zwp_linux_dmabuf_v1`. Compositor imports the buffer and
scanouts directly (or composites via GPU). Zero-copy for fullscreen clients.

### Input Events

The compositor sends input events to focused clients via the Wayland protocol:

| Object | Events |
|--------|--------|
| `wl_keyboard` | `keymap`, `enter`, `leave`, `key`, `modifiers` |
| `wl_pointer` | `enter`, `leave`, `motion`, `button`, `axis` |
| `wl_seat` | `capabilities`, `name` |

These replace Lepton's `OP_KEYBOARD_INPUT` / `OP_MOUSE_INPUT` messages. The
compositor reads `/dev/kbd` and `/dev/mouse` and translates to Wayland events.

## Frame vs Service

| Component | Layer | Reason |
|-----------|-------|--------|
| Wayland wire protocol | Userspace | libwayland-client handles marshalling |
| Compositor window management | Userspace | Pure Rust application |
| Surface compositing | Userspace | Pixel buffer blitting |
| Display backend (fb0 mmap) | Userspace | Uses existing `/dev/fb0` interface |
| DMA-buf import (Phase 2) | Kernel + Userspace | Kernel manages buffer objects |

## Dependencies

- **Unix Domain Sockets**: Wayland transport (new).
- **Display Infrastructure**: `/dev/fb0` mmap and ioctl (complete).
- **Input Handling**: `/dev/kbd` and `/dev/mouse` (complete).
- **IPC Channels**: fd-passing for shm pool buffers (complete).
- **Mesa & Vulkan**: WSI is the consumer of this protocol (parallel work).
